# 05 — Eval-driven prompt iteration

**Industry name(s):** eval-driven development / golden sets / LLM-as-judge.
**Type:** Industry standard.

## Zoom out, then zoom in

This is the dividing line between junior and senior prompt work. A junior
iterates by vibes — "the response feels better now." A senior iterates against an
eval set and keeps a change only if the score improved without regressing a case
that mattered. AptKit's whole testing backbone is built for this. Look at where
it sits.

```
  Zoom out — where evals close the prompt loop

  ┌─ Prompt/Agent layer ────────────────────────────────────────┐
  │  change a prompt → run agent → produce output                │
  └───────────────────────────┬──────────────────────────────────┘
                             │  saved as a replay artifact
  ┌─ Eval layer (packages/evals) ─▼─────────────────────────────┐
  │  ★ structural-diff · detection-scorer · rubric-judge ★       │ ← we are here
  │     replay-runner: batch eval → ReplayArtifactEvalSummary    │
  └───────────────────────────┬──────────────────────────────────┘
                             │  promote passing runs
  ┌─ Fixture layer ──────────▼──────────────────────────────────┐
  │  fixtures/promoted/*.json → deterministic FixtureModelProvider│
  └───────────────────────────────────────────────────────────────┘
```

Now zoom in. The pattern: live run → artifact → eval → promote to fixture →
deterministic replay. The promoted fixtures *are* the regression suite — they're
correctness baselines, regenerated by tooling, never hand-edited. You write the
eval before you iterate the prompt, so "better" means something measurable.

## Structure pass

**Layers.** Three: *graded* outputs (an artifact scored by a scorer), the *batch*
runner (`replay-runner` walks every artifact), and the *baseline* (promoted
fixtures replayed deterministically).

**Axis — held constant: "what decides if this output is good?"**

```
  One question down the eval stack: what's the judge?

  ┌─ structural-diff ─────────┐  → SHAPE rules (deterministic)
  ┌─ detection-scorer ────────┐  → SET membership: required categories matched?
  ┌─ rubric-judge ────────────┐  → an LLM, scoring against a rubric
  ┌─ promoted fixture replay ─┐  → EXACT recorded output (regression baseline)
```

**Seam — the promote step.** The load-bearing seam is "live run → promoted
fixture." Above it, output is non-deterministic (a real model). Below it, it's
frozen and replayed exactly. The axis flips from "model decides" to "recorded
truth decides." That's why editing a promoted fixture changes test meaning — you'd
be editing the baseline, not the prompt.

## How it works

#### Move 1 — the mental model

You already write tests before refactoring: the test pins the behavior, then you
change the code freely because the test catches a regression. An eval set is the
test; the prompt is the code. You write the eval first so a prompt change is safe
to make.

```
  The eval loop — change behind a fixed measure

  golden set (cases + expectations)
        │  written FIRST
        ▼
  change prompt ──► run ──► score ──► improved & no regression?
        ▲                              │ yes            │ no
        │                              ▼                ▼
        └───────────── keep ────── promote to fixture   discard
```

#### Move 2 — the walkthrough

**The graded output — three scorer styles.** AptKit grades by output shape, not
one universal metric:

- `structural-diff` — deterministic shape rules (is the artifact the right
  shape?).
- `detection-scorer` — set membership for monitoring: did the run produce the
  required categories / metrics / scopes / severities, within a count band?
  Returns matched / missed / unexpected and a score.
- `rubric-judge` — an LLM scoring a subject against a rubric's dimensions
  (LLM-as-judge; see below).

**Breaks if missing:** without a scorer you're back to vibes — you can't tell a
prompt edit improved anything.

```
  detection-scorer — set membership as a score

  expected: requiredCategories = ['payment_mix_shift', 'revenue_drop']
  produced: ['payment_mix_shift', 'cart_abandonment']
        │
        ▼
  matched = ['payment_mix_shift']  missed = ['revenue_drop']
  unexpected = ['cart_abandonment']  → score reflects the gap
```

**LLM-as-judge — `rubric-judge`.** For outputs that can't be checked by shape
(was this diagnosis well-reasoned?), a model scores against a structured rubric:
each dimension has a labeled 1–N scale, the judge returns a score + reason per
dimension plus a verdict and one fix. Critically, the judge's output is itself
validated — scores must fall in each dimension's declared range, the verdict must
be an allowed value. **When appropriate:** open-ended quality where a human review
is the only alternative. **The trap:** a model judging output has the same blind
spots that produced it; calibration examples anchor the scale to fight drift.

```
  rubric-judge — LLM scores, code validates the judge

  subject → judge prompt (dimensions + scale + verdicts + calibration examples)
        │
        ▼  model emits { dimensions:{id:{score,reason}}, verdict, fix }
  validate: every score in [min,max]? verdict allowed? → typed RubricJudgment
        │
        └─ the judge is not trusted either — out-of-range scores are rejected
```

**The batch runner — `replay-runner`.** Walks every artifact in a directory in
deterministic filename order, asserts each one's shape, and rolls up a report:
checked, failed, per-artifact summary. **Breaks if missing:** you eval one case by
hand and miss the regression in case #14.

**The baseline — promoted fixtures.** A passing live run is promoted to a
timestamped fixture and replayed deterministically by `FixtureModelProvider`. This
is the regression suite: production behavior frozen so a future prompt change that
breaks it fails a test. **Breaks if missing:** the classic eval failure — a
"better" prompt that lifts the average but silently regresses a critical edge case
nobody re-ran.

#### Move 3 — the principle

Write the eval before you iterate the prompt. Skipping evals isn't faster — it's
slower, because you iterate in circles, unable to tell improvement from
regression. The promoted fixture is the part people forget: an eval you run once
proves nothing; an eval that becomes a permanent regression baseline is what makes
the next change safe.

## Implementation in codebase

**Use cases.** Monitoring outputs are graded with `detection-scorer` (did it find
the right anomalies?). The rubric judge grades open-ended quality. `replay-runner`
batch-evals every saved artifact. Promoted fixtures are the per-agent regression
baselines replayed in `node --test`.

The LLM-judge prompt — dimensions, scale, verdicts, calibration, and a strict
output shape:

```
  packages/evals/src/rubric-judge.ts  (lines 143–160)

    'Never rewrite the subject. Return one highest-leverage fix, not a list.',
    'Rubric dimensions:', dimensions,
    'Allowed verdicts:', verdicts,
```

## Elaborate

Hamel Husain's writing is the canonical reference here — the discipline of golden
sets, error analysis, and "look at your data" before reaching for a fancier model.
AptKit's replay-centric design is a clean instantiation of it: the artifact is the
unit of evaluation, the promoted fixture is the golden case made permanent, and
the three scorers cover the three shapes of output (structured, set-membership,
open-ended).

The honest gaps: there's no explicit "hand-curated 20–50 case golden set"
directory — the golden set *emerges* from promoted fixtures rather than being
authored up front, which means coverage depends on what got promoted. And the
LLM-judge has the diminishing-returns problem the spec names: a model judging a
model shares blind spots; the calibration examples mitigate but don't remove it.
The `rubric-improvement` agent isn't wired into the root `replay:promoted`
pipeline (per context.md), so its regression baseline isn't run automatically —
that's a gap to close.

Where it connects: 02 (a rising schema-fail rate is exactly what a batch eval
should surface), 03 (record the prompt `version` in the artifact so the eval diff
attributes a regression to a prompt change vs a model upgrade), and 04 (token/cost
belong in the eval summary so you catch a chain that got more expensive).

## Interview defense

**Q: How do you iterate on a prompt without breaking what worked?**
Write the eval first, then change the prompt behind it. Live run → artifact →
score → promote passing runs to fixtures replayed deterministically. "Better"
means the score improved with no regression on a promoted case. The promoted
fixture is the part people skip — it's what turns a one-time eval into a permanent
regression guard.

```
  golden set (written first) → change prompt → score → improved & no regress?
        ▲                                                   │
        └──────────── promote to fixture (baseline) ◄──────┘
```
Anchor: "replay → eval → promote, `replay-runner.ts:46`; promoted fixtures are
never hand-edited (context.md)."

**Q: When is LLM-as-judge appropriate, and what's its failure mode?**
For open-ended quality you can't check by shape — was this diagnosis
well-reasoned? Score against a structured rubric with a fixed scale and validate
the judge's output (scores in range, verdict allowed). The failure mode: the judge
shares the generator's blind spots. Calibration examples anchor the scale to fight
that; they don't remove it.
Anchor: "`rubric-judge.ts:143` builds the prompt; `:195` validates the score range."
