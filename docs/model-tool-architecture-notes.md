# Model And Tool Architecture Notes

## Core Design Decision

AptKit separates three responsibilities that are often coupled in AI application code:

- `model provider`: the system that reasons and generates model output.
- `tool registry`: the catalog of callable tools the model is allowed to request.
- `tool implementation`: the actual execution path for a tool request.

This lets the same capability run against different model providers and different tool backends without changing the capability logic.

## Current Studio Modes

### Fixture

`Fixture` mode uses:

- Fake/scripted model responses from JSON fixtures.
- Fake/in-memory tool results from JSON fixtures.
- No network.
- No API keys.
- Deterministic output on every run.

Use this mode for UI work, regression tests, trace rendering, fixture replay, and eval wiring.

### Anthropic

`Anthropic` mode uses:

- Real Anthropic model calls.
- The same in-memory fixture tools.
- Server-side `ANTHROPIC_API_KEY`.
- Non-deterministic model output, but deterministic tool data.

Use this mode to test real Claude reasoning, tool choice, prompt behavior, JSON output quality, and recovery behavior without debugging live data adapters.

### OpenAI

`OpenAI` mode uses:

- Real OpenAI model calls.
- The same in-memory fixture tools.
- Server-side `OPENAI_API_KEY`.
- Non-deterministic model output, but deterministic tool data.

Use this mode to compare OpenAI behavior against Anthropic on the same capability, prompts, tool catalog, and fixture data.

## Fake Tools With Real Models

In live provider modes, the tools are not fake to the model. They are fake to AptKit.

The model experiences a normal tool loop:

```txt
model: call list_voucher_pools
AptKit: here is the result: { data: [] }
model: use that result to write recommendations
```

AptKit advertises the fixture tool catalog to the model. When the model calls a tool, AptKit intercepts that call and returns deterministic fixture data instead of calling a real external system.

This tests the model's reasoning over tools while keeping the data safe, cheap, and repeatable.

## Testing Ladder

AptKit should move capabilities through this ladder:

1. `fake model + fake tools`

Deterministic regression testing. Good for package tests, UI rendering, trace formatting, and eval assertions.

2. `real model + fake tools`

Provider behavior testing. Good for comparing Anthropic vs OpenAI, prompt quality, tool selection, JSON validity, and recovery behavior.

3. `real model + real tools`

Integration testing and production readiness. Good for validating live data adapters, auth, rate limits, latency, error handling, and real-world tool outputs.

Another way to think about the same ladder:

```txt
Fixture model + fake tools
= deterministic replay / regression test

Real model + fake tools
= test reasoning quality safely

Real model + real tools
= production-style agent behavior
```

When Studio is in `OpenAI` mode, AptKit is using live OpenAI-generated model output. The recommendation text is generated by OpenAI at run time, while the business data and tool results still come from controlled JSON fixtures.

That makes `OpenAI` mode a middle stage: real model behavior, but safe fake data. It is useful before connecting the capability to real tools or production data.

## Why This Matters

This architecture keeps each failure mode isolated:

- If fixture mode fails, the bug is in capability logic, parsing, trace handling, evals, or UI.
- If live-model fixture-tool mode fails, the bug is likely prompt/provider behavior, model output shape, or provider adapter mapping.
- If live-model live-tool mode fails, the bug may be in auth, external APIs, data shape drift, rate limits, network behavior, or adapter logic.

The result is a safer extraction path from app-specific AI code into reusable packages.

## Design Principle

Do not let agents import provider SDKs or call external tools directly.

Capabilities should depend on:

- `ModelProvider`
- `ToolRegistry`
- typed input/output schemas
- trace sinks
- optional abort signals

Provider SDKs belong in provider adapter packages. External tool integrations belong in tool or adapter packages. The capability stays focused on behavior.
